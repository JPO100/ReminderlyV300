## Executive summary

Reminderly is a functional, well-featured reminders app with a sophisticated NLC system, comprehensive dev self-checks (477 checks), and clean separation between reminders and lists. The core logic (NLC parser, interaction, normalisation, categorisation, notifications) is solid and well-tested through the self-check system.

The primary risk to the project is concentration. Almost all state and logic lives in a single 5,794-line App.tsx with 60+ state variables. This does not break the app today, but it makes every future change riskier and harder for both humans and AI assistants to modify safely.

Secondary risks are accumulation of dead code (45+ unused files in `src/imports/`, 44 of 46 unused shadcn/ui components), a stale self-check baseline, and silent failure on localStorage operations.

The app is not fragile. It is functional and shippable. The issues below are about reducing risk for ongoing development, not about fixing a broken product.

## Must fix

### 1. Stale self-check baseline

- Why it matters: the self-check system is the project's primary quality gate. BASELINE.md reports 280 checks but the current suite has 477. This means 197 checks have no documented expected output. If a check starts failing, there is no way to know whether it was always failing or is a regression.
- Files involved: `src/app/dev/BASELINE.md`
- Recommended fix: re-run the full self-check suite via DevTools overlay, copy the output, and replace BASELINE.md content
- Complexity: low

### 2. Notification ID hash collision risk

- Why it matters: notification IDs are generated by hashing reminder IDs with a simple `(acc * 31 + charCode) % 2147483647` function (notifications.ts line 133). This hash has no collision detection. Two different reminder IDs could produce the same notification ID, causing one notification to silently overwrite another. With the 64-notification cap, this could mean a user misses a reminder notification entirely.
- Files involved: `src/app/notifications.ts` (lines 133-136)
- Recommended fix: add a collision detection step after hashing. If a collision is detected, increment the ID by 1 until unique. Alternatively, use a more robust hash function. Keep it simple - a post-hash collision check is sufficient.
- Complexity: low

### 3. localStorage persistence fails silently

- Why it matters: all reminder and list data is persisted to localStorage. If localStorage is full, in private browsing mode, or otherwise unavailable, every persistence effect in App.tsx (there are 20+ of them) silently catches and ignores the error. The user would continue using the app thinking their data is saved, then lose everything on refresh.
- Files involved: `src/app/App.tsx` (lines 1337-1603, all persistence effects)
- Recommended fix: add a single shared persistence helper that logs failures and sets a visible error state. Display a minimal warning banner when persistence fails. Do not over-engineer - a simple boolean state like `persistenceError` and a small banner is sufficient.
- Complexity: low-medium

## Anti-stall notes for future prompts

### 1. App.tsx size causes wrong-location edits

- Risk: at 5,794 lines with 60+ state variables, 30+ useEffects, and 20+ handlers, App.tsx is too large for AI assistants to hold fully in context. This leads to: edits in the wrong location, duplicate state additions, missed cleanup effects, broken dependency arrays.
- Files involved: `src/app/App.tsx`
- Why this causes problems: Claude reads App.tsx in chunks. When asked to add or modify a feature, it may not see all related state, effects, and handlers. It may add a new state variable without seeing an existing one that serves the same purpose, or modify a handler without updating the corresponding cleanup effect.
- Recommended prompt wording: always specify the exact line range or function name. Example: "In App.tsx, modify the handleCompleteClick function (around line 2675) to also do X." Do not say "add completion logic to App.tsx" without specifying where. When adding state, always ask Claude to first search for existing related state.

### 2. NewReminderOverlay NLC integration is fragile to modifications

- Risk: NewReminderOverlay.tsx (1,290 lines) has a complex mirror+hit layer system for NLC token display, with refs mirroring toggle state to avoid stale closures, and eslint-disable-line comments on critical dependency arrays. Changes to NLC behaviour or toggle logic can easily break the interaction pipeline.
- Files involved: `src/imports/NewReminderOverlay.tsx`, `src/app/utils/nlc-parser.ts`, `src/app/utils/nlc-interaction.ts`
- Why this causes problems: the NLC system has three layers (parsing, interaction, overlay rendering) that must stay synchronised. A change to token parsing can break interaction logic. A change to toggle state refs can cause stale closures. The eslint-disable-line comments hide genuine dependency issues.
- Recommended prompt wording: "When modifying NLC behaviour, specify which layer to change: parser (nlc-parser.ts), interaction logic (nlc-interaction.ts), or overlay rendering (NewReminderOverlay.tsx). Do not change multiple layers in one step. After any change, specify whether the self-checks for that layer should still pass."

### 3. src/imports/ naming causes wrong-file edits

- Risk: `src/imports/` contains files with names that overlap with active components. For example: `app.tsx` (specification, not the app), `DevToolsLogin.tsx` (dead code, not the active dev tools), `reminder-overlay-patch.tsx` (patch, not the overlay). Claude may edit these instead of the correct file.
- Files involved: `src/imports/app.tsx`, `src/imports/app-revert.tsx`, `src/imports/app-fix.tsx`, `src/imports/DevToolsLogin.tsx`, `src/imports/reminder-overlay-patch.tsx`
- Why this causes problems: when asked to "modify the app" or "fix the dev tools login", Claude may find and edit a file in `src/imports/` that is not the active code. This leads to changes that compile but have no effect.
- Recommended prompt wording: always use full paths. Example: "Edit src/app/App.tsx" not "edit the app". "Edit src/app/components/DevToolsOverlay.tsx" not "edit the dev tools". Removing dead code from `src/imports/` is the best long-term fix.

### 4. Multiple timer ref patterns create cleanup risk

- Risk: App.tsx uses 15+ timer ref patterns (`completionTimersRef`, `uncompleteTimersRef`, `pendingDeleteTimersRef`, `undeleteTimersRef`, `rescheduleTimersRef`, `newReminderInsertTimerRef`, `insertHighlightTimerRef`, `listInsertTimerRef`, `listInsertHighlightTimerRef`, `listItemInsertHighlightTimerRef`, `clearListTimerRef`, `alphabeticalListItemTimerRef`, plus list-specific variants). Each has a corresponding cleanup useEffect. When adding new timed behaviour, failing to add the cleanup effect creates a memory leak.
- Files involved: `src/app/App.tsx` (lines 1230-1334 for cleanup effects)
- Why this causes problems: the pattern is correct but not documented. A new timer added without its cleanup effect will not break anything immediately but will leak on component unmount. Claude cannot infer the need for cleanup by reading a subset of the file.
- Recommended prompt wording: "When adding any new setTimeout or setInterval in App.tsx, also add a cleanup useEffect that clears the timer on unmount. Follow the pattern at line 1230 (cleanup overlay done timer) as a template."

### 5. Feature flag proliferation makes testing complex

- Risk: there are 10+ feature flags persisted in localStorage, each with its own useState and persistence useEffect. The interaction between flags creates a combinatorial explosion of app states. For example: `isListsEnabled` gates lists, but also affects `smartRemindersFeatureEnabled`, `savedListsFeatureEnabled`, and tutorial behaviour. Changing one flag without understanding the cascade can produce unexpected states.
- Files involved: `src/app/App.tsx` (lines 1026-1132 for flag definitions, lines 1359-1445 for persistence), `src/app/components/DevToolsOverlay.tsx` (flag toggle UI)
- Why this causes problems: Claude may toggle a flag without realising it has downstream effects. The flag dependency chain is implicit (enforced in effects, not in types).
- Recommended prompt wording: "When modifying a feature flag, check what other flags or effects depend on it. The key dependency is: lists feature gates smart reminders (line 1531-1535). Specify whether the flag change should cascade or be isolated."

## Nice to fix

### 1. Package.json name is inherited scaffolding

- Why it matters: `package.json` has `"name": "@figma/my-make-file"` which is the default name from the Figma code generation tool. This has no functional impact but is confusing in logs, error messages, and any tooling that reads the package name.
- Files involved: `package.json` (line 2)

- Complexity: trivial

### 2. Hard-coded fallback date in list-utils.ts

- Why it matters: `formatListDueDate()` returns `'Jan 1st 2026'` as a fallback when date parsing fails (line 43). This is a defensive fallback that should never be reached, but if it is, the user sees a specific incorrect date rather than a clear error state like "No due date".
- Files involved: `src/app/utils/list-utils.ts` (line 43)
- Recommended fix: change fallback to an empty string or "No date"
- Complexity: trivial

### 3. useNotificationTapHandler re-subscribes on every reminder change

- Why it matters: the effect in `useNotificationTapHandler.ts` depends on the full `reminders` array and all callback props. This means the Capacitor event listener is removed and re-added every time any reminder changes. Functionally correct but unnecessary work.
- Files involved: `src/app/useNotificationTapHandler.ts` (lines 84-97)
- Recommended fix: use refs for the callbacks and reminders array so the effect only subscribes once. Low priority since the current pattern works.
- Complexity: low

### 4. Dev tools password is a hard-coded string

- Why it matters: the DevTools login screen uses `"GILBURN"` as the password (DevToolsOverlay.tsx line 895). This is fine for a dev-only tool but means anyone reading the source can access dev tools. Since this is a hybrid app with bundled JS, the password is visible in the built assets.
- Files involved: `src/app/components/DevToolsOverlay.tsx` (line 895)
- Recommended fix: acceptable as-is for a dev tool behind a 3-click unlock. If concerned, the `isDevToolsPasswordRequired` flag can gate access independently. No action needed unless dev tools expose user data in production builds.
- Complexity: n/a (informational)

### 5. Midnight badge notification uses fragile +1ms offset

- Why it matters: in `notifications.ts` (line 93), badge count at midnight is computed with `new Date(midnight.getTime() + 1)` to avoid a boundary edge case. This works but is a brittle hack that could fail if the system clock is imprecise or if midnight calculation drifts.
- Files involved: `src/app/notifications.ts` (line 93)
- Recommended fix: use the next day's date at 00:00:00 explicitly rather than adding 1ms to midnight
- Complexity: trivial

## Leave alone

### 1. App.tsx single-component architecture

App.tsx is large (5,794 lines) and this is noted as an anti-stall risk. However, the current architecture works and is well-understood. Do not split it into smaller components, add context providers, or introduce state management libraries without a specific, motivated reason. Splitting for the sake of splitting creates new coordination risks and breaks the existing mental model. The anti-stall mitigations (precise prompt wording, line number references) are sufficient for now.

### 2. NLC three-layer system

The NLC system (parser, interaction, overlay) is complex but correct and well-tested with 109 self-checks. The mirror+hit layer pattern in NewReminderOverlay.tsx with ref-mirrored toggle state is unusual but necessary to avoid stale closures in the token interaction pipeline. Do not simplify or refactor this unless a specific bug motivates it.

### 3. Timer ref pattern in App.tsx

The 15+ timer ref pattern with individual cleanup effects is verbose but safe. All timers are properly cleaned up. Do not consolidate these into a single "timer manager" abstraction - the current pattern is explicit and each timer's lifecycle is independently visible.

### 4. Onboarding flow (9 pages)

Nine onboarding pages (OnboardingPage1Content through OnboardingPage9Content) is a lot of content, but each page is simple (30-50 lines) and self-contained. Do not consolidate into a data-driven system unless the content needs frequent changes. The current pattern is easy to edit one page at a time.

### 5. Self-check system architecture

The 477-check system in `src/app/dev/` using exported functions with `withIsolatedStorage()` is a good fit for this project. It runs in-browser via DevTools overlay and provides comprehensive coverage without the overhead of a full test framework integration. Do not migrate to vitest or another test runner unless CI/CD integration becomes a requirement.

### 6. Existing localStorage key conventions

The mix of key formats (`reminderly.reminders.v1`, `reminderly-created-lists`, `dev.listsEnabled`, `reminderly-ff-*`) evolved over time. Renaming keys would require migration logic for existing users. Leave the naming as-is.

### 7. Reminder and list structural separation

Reminders and lists are stored, rendered, and managed as completely separate systems bridged only by smart reminders (`isSmartReminder` + `linkedListId`). This separation is correct and intentional. Do not merge their data models, storage, or state management.
